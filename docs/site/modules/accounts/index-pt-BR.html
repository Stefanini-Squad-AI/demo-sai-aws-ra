📦 Módulo de Contas (Accounts)

Guia de Desenvolvimento de User Stories

Propósito: Gestão completa do ciclo de vida das contas de cartões de crédito, incluindo visualização financeira e atualização transacional dos dados pessoais dos clientes

📋 Resumo do Módulo

Contexto de Negócio: Este módulo permite que representantes de atendimento e administradores gerenciem contas de cartões de crédito, desde consultas de saldo até atualizações de limites e dados pessoais dos clientes.

Responsabilidades principais:

  • Busca e visualização de contas por ID de 11 dígitos
  • Exibição de informações financeiras (limites, saldos, ciclos)
  • Atualização transacional de dados da conta e do cliente
  • Proteção de dados sensíveis (SSN, números de cartão)
  • Validações de negócio em tempo real

🎯 Guia de Desenvolvimento de User Stories

📝 Templates de US Específicos

Exemplo: Como administrador de contas, quero atualizar o limite de crédito para refletir a melhora no score FICO do cliente

Validação e Segurança

Padrão: Como [usuário], quero [validações automáticas] para [evitar erros]

Exemplo: Como oficial de compliance, quero que o SSN seja sempre exibido mascarado para atender às normas PCI-DSS

⚡ Fatores de Aceleração do Desenvolvimento

  • Hook useAccountView: Hook reutilizável que encapsula busca e controle de estado. Acelera ao evitar reimplementar lógica de carregamento e erros.
  • Hook useAccountUpdate: Detecta automaticamente alterações comparando estado original e atual. Reduz a complexidade de formulários de edição.
  • AccountValidationService (Backend): Validações centralizadas evitam duplicidade de código. Novas regras são adicionadas uma única vez.
  • Componentes Material-UI: TextField, Card e Button pré-configurados reduzem o tempo de UI em ~60%.
  • API RESTful documentada: 4 endpoints bem definidos facilitam a integração frontend-backend em 1-2 dias.

  • Complexo (5-8 pts): Integrar com sistema externo de scoring, implementar auditoria completa de alterações

🏗️ Componentes-Chave

AccountViewScreen.tsx

Responsabilidade: Tela de busca e visualização de contas com 4 cards de informação

Reutilização: Funções de mascaramento (maskSSN, maskCard) exportáveis para outros módulos

AccountUpdateScreen.tsx

Responsabilidade: Edição com modo toggle, confirmação de salvamento e detecção de alterações

Reutilização: Padrão de toggle edit mode aplicável a qualquer formulário

useAccountView.ts

Responsabilidade: Hook de busca com validação inline e gerenciamento de estados (loading/error/data)

Reutilização: Modelo para criar hooks de busca em outros módulos

useAccountUpdate.ts

Responsabilidade: Hook de atualização com detecção automática de alterações

Reutilização: Lógica reutilizável de comparação JSON

AccountViewService.java

Responsabilidade: Busca multi-entidade (CardXref → Account → Customer) com mascaramento de SSN

Reutilização: Padrão de leitura de três entidades relacionadas

AccountUpdateService.java

Responsabilidade: Atualização transacional atômica de Account + Customer

Reutilização: Padrão @Transactional para operações multi-entidade

AccountValidationService.java

Responsabilidade: Validações de negócio centralizadas (FICO, ZIP, nomes, etc.)

Reutilização: Métodos reutilizáveis (isValidAccountId, validateYesNo, etc.)

🔗 Dependências

  • Account Entity: Entidade JPA com 11 campos financeiros e operacionais
  • Customer Entity: Entidade JPA com 14 campos pessoais e de contato
  • CardXrefRecord: Tabela de relacionamento Account-Customer-Card
  • Material-UI: TextField, Card, Button, IconButton, Snackbar
  • Redux Toolkit: Gerenciamento de estado global (opcional para este módulo)

🌐 Interfaces Públicas (APIs)

GET  /api/account-view?accountId={id}
     → Buscar e obter conta completa com dados do cliente
     → Response: AccountViewResponseDto (JSON)

GET  /api/account-view/initialize
     → Inicializar a tela com metadata do sistema
     → Response: AccountViewResponseDto com data/hora/transactionId

GET  /api/accounts/{accountId}
     → Obter dados da conta para o formulário de edição
     → Response: AccountUpdateRequestDto (JSON)

PUT  /api/accounts/{accountId}
     → Atualizar conta e cliente (transacional)
     → Body: AccountUpdateRequestDto
     → Response: Map<String, String> com mensagem de sucesso

📋 Regras de Negócio

Identificação e Busca

RN-001: O Account ID deve ter exatamente 11 dígitos numéricos
RN-002: O Account ID não pode ser composto apenas por zeros (00000000000)
RN-003: A busca valida a existência em: CardXrefRecord → Account → Customer

Segurança e Privacidade

RN-006: O SSN é exibido mascarado: ***-**-XXXX (apenas os últimos 4 dígitos)
RN-007: O número do cartão é mascarado: ****-****-****-XXXX

Validações Críticas

RN-009: Active Status só aceita 'Y' (ativo) ou 'N' (inativo)
RN-012: FICO Score deve estar no intervalo 300-850
RN-015: ZIP Code no formato: ^\d{5}(-\d{4})?$

Transacionalidade

RN-018: Atualização de Account e Customer é atômica (tudo ou nada)
RN-021: Executa-se "READ FOR UPDATE" (lock pessimista) antes de atualizar

⚡ Considerações de Performance

  • Tempo de Resposta: < 500ms (P95) para buscas
  • Throughput: Suporta 100 buscas concorrentes/segundo
  • Uso de Memória: < 50MB por sessão de usuário
  • Estratégia de Cache: Sem cache atualmente (queries diretas ao PostgreSQL)
  • Índices DB: Necessários em accountId, customerId e cardNumber

🚨 Riscos de Desenvolvimento e Mitigações

⚠️ Performance nas Buscas

Risco: Busca sequencial em 3 tabelas pode degradar com alto volume

Mitigação: Criar índices em accountId e customerId; considerar cache Redis para contas frequentes

⚠️ Ausência de i18n

Risco: Mensagens hardcodeadas em inglês não atendem aos requisitos de idioma

Mitigação: Implementar react-i18next antes de novas funcionalidades (estimado 5 pontos)

⚠️ Sem Auditoria de Alterações

Risco: Não há rastreabilidade de quem alterou quais dados

Mitigação: Implementar Audit Trail com Spring Data Envers (5 pontos)

⚠️ Validações COBOL Comentadas

Risco: Validação do SSN está comentada no código migrado

Mitigação: Revisar e ativar ou substituir por validações modernas

Padrão de Implementação: Hook de Busca

// useAccountView.ts - Padrão reutilizável
export function useAccountView() {
  const [data, setData] = useState<AccountViewResponse | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const searchAccount = async (request: AccountViewRequest) => {
    setLoading(true);
    try {
      const accountIdNum = parseInt(request.accountId);
      if (isNaN(accountIdNum) || accountIdNum === 0) {
        throw new Error("Account must be non-zero 11 digit number");
      }
      
      const response = await api.get<AccountViewResponse>(
        `/account-view?accountId=${request.accountId.padStart(11, "0")}`
      );
      setData(response.data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  return { data, loading, error, searchAccount };
}

Padrão Backend: Serviço Transacional

// AccountUpdateService.java
@Service
public class AccountUpdateService {
  
  @Transactional  // ← Chave: garante atomicidade
  public Map<String, String> updateAccountAndCustomer(
      Long accountId, AccountUpdateRequestDto request) {
    
    // Ler com lock
    Account account = accountRepository.findById(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Account not found"));
    Customer customer = customerRepository.findByAccountId(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Customer not found"));
    
    // Validar
    List<String> errors = validationService.validateAccountUpdate(request);
    if (!errors.isEmpty()) {
      throw new ValidationException(String.join(", ", errors));
    }
    
    // Atualizar
    updateAccountFromRequest(account, request);
    updateCustomerFromRequest(customer, request);
    
    // Salvar (se falhar, rollback automático)
    accountRepository.save(account);
    customerRepository.save(customer);
    
    return Map.of("message", "Account updated successfully");
  }
}
