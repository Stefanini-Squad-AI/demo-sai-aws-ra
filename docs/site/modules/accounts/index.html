📦 Módulo de Cuentas (Accounts)

Guía de Desarrollo de Historias de Usuario

Propósito: Gestión completa del ciclo de vida de cuentas de tarjetas de crédito, incluyendo visualización de información financiera y actualización transaccional de datos

Español (original) Português (pt-BR)

📋 Resumen del Módulo

Contexto de Negocio: Este módulo permite a representantes de servicio al cliente y administradores gestionar cuentas de tarjetas de crédito, desde consultas de saldo hasta actualizaciones de límites y datos personales de clientes.

Responsabilidades Principales:

  • Búsqueda y visualización de cuentas por ID de 11 dígitos
  • Mostrar información financiera (límites, balances, ciclos)
  • Actualización transaccional de datos de cuenta y cliente
  • Protección de datos sensibles (SSN, números de tarjeta)
  • Validaciones de negocio en tiempo real

⚡ Factores de Aceleración de Desarrollo

  • useAccountView Hook: Hook reutilizable que encapsula búsqueda y manejo de estado. Acelera desarrollo al evitar reimplementar lógica de carga y errores.
  • useAccountUpdate Hook: Detecta cambios automticamente comparando estado original vs actual. Reduce complejidad de formularios de edición.
  • AccountValidationService (Backend): Validaciones centralizadas evitan código duplicado. Nuevas validaciones se agregan una sola vez.
  • Material-UI Components: TextField, Card, Button preconstruidos reducen tiempo de UI en 60%.
  • API RESTful Documentada: 4 endpoints bien definidos facilitan integración frontend-backend en 1-2 días.

Responsabilidad: Búsqueda multi-entidad (CardXref → Account → Customer) con enmascarado de SSN

Reusabilidad: Patrón de lectura de 3 entidades relacionadas

AccountUpdateService.java

Responsabilidad: Actualización transaccional atómica de Account + Customer

Reusabilidad: Patrón @Transactional para operaciones multi-entidad

AccountValidationService.java

Responsabilidad: Validaciones de negocio centralizadas (FICO, ZIP, nombres, etc.)

Reusabilidad: Métodos de validación reutilizables (isValidAccountId, validateYesNo, etc.)

🔗 Dependencias

  • Account Entity: JPA Entity con 11 campos financieros y operativos
  • Customer Entity: JPA Entity con 14 campos personales y de contacto
  • CardXrefRecord: Tabla de relación Account-Customer-Card
  • Material-UI: TextField, Card, Button, IconButton, Snackbar
  • Redux Toolkit: Gestión de estado global (opcional para este módulo)

🌐 Interfaces Públicas (APIs)

GET  /api/account-view?accountId={id}
     → Buscar y obtener cuenta completa con datos de cliente
     → Response: AccountViewResponseDto (JSON)

GET  /api/account-view/initialize
     → Inicializar pantalla con metadata del sistema
     → Response: AccountViewResponseDto con fecha/hora/transactionId

GET  /api/accounts/{accountId}
     → Obtener datos de cuenta para formulario de edición
     → Response: AccountUpdateRequestDto (JSON)

PUT  /api/accounts/{accountId}
     → Actualizar cuenta y cliente (transaccional)
     → Body: AccountUpdateRequestDto
     → Response: Map<String, String> con mensaje de éxito

📋 Reglas de Negocio

Identificación y Búsqueda

RN-001: El Account ID debe ser exactamente de 11 dígitos numéricos
RN-002: El Account ID no puede ser todo ceros (00000000000)
RN-003: La búsqueda valida existencia en: CardXrefRecord → Account → Customer

Seguridad y Privacidad

RN-006: El SSN se muestra enmascarado: ***-**-XXXX (solo últimos 4 dígitos)
RN-007: El número de tarjeta se enmascara: ****-****-****-XXXX

Validaciones Críticas

RN-009: Active Status solo acepta 'Y' (activo) o 'N' (inactivo)
RN-012: FICO Score debe estar en rango 300-850
RN-015: ZIP Code formato: ^\d{5}(-\d{4})?$

Transaccionalidad

RN-018: Actualización de Account y Customer es atómica (todo o nada)
RN-021: Se hace "READ FOR UPDATE" (lock pesimista) antes de actualizar

⚡ Consideraciones de Rendimiento

  • Tiempo de Respuesta: < 500ms (P95) para búsquedas
  • Throughput: Soporta 100 búsquedas concurrentes/segundo
  • Uso de Memoria: < 50MB por sesión de usuario
  • Estrategia de Caché: Sin caché actualmente (queries directas a PostgreSQL)
  • Índices DB: Requeridos en accountId, customerId, cardNumber

🚨 Riesgos de Desarrollo y Mitigaciones

⚠️ Performance en Búsquedas

Riesgo: Búsqueda secuencial en 3 tablas puede degradarse con alto volumen

Mitigación: Implementar índices en accountId y customerId; considerar caché Redis para cuentas frecuentes

⚠️ Falta de i18n

Riesgo: Mensajes hardcodeados en inglés no cumplen requisito de español

Mitigación: Implementar react-i18next antes de nuevas funcionalidades (estimado 5 puntos)

⚠️ Sin Auditoría de Cambios

Riesgo: No hay trazabilidad de quién modificó qué valores

Mitigación: Implementar Audit Trail con Spring Data Envers (5 puntos)

⚠️ Validaciones COBOL Comentadas

Riesgo: Validación de SSN está comentada en código migrado

Mitigación: Revisar y habilitar o reemplazar con validaciones modernas

Patrón de Implementación: Hook de Búsqueda

// useAccountView.ts - Patrón reutilizable
export function useAccountView() {
  const [data, setData] = useState<AccountViewResponse | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const searchAccount = async (request: AccountViewRequest) => {
    setLoading(true);
    try {
      const accountIdNum = parseInt(request.accountId);
      if (isNaN(accountIdNum) || accountIdNum === 0) {
        throw new Error("Account must be non-zero 11 digit number");
      }
      
      const response = await api.get<AccountViewResponse>(
        `/account-view?accountId=${request.accountId.padStart(11, "0")}`
      );
      setData(response.data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  return { data, loading, error, searchAccount };
}

Patrón Backend: Servicio Transaccional

// AccountUpdateService.java
@Service
public class AccountUpdateService {
  
  @Transactional  // ← Clave: garantiza atomicidad
  public Map<String, String> updateAccountAndCustomer(
      Long accountId, AccountUpdateRequestDto request) {
    
    // Leer con lock
    Account account = accountRepository.findById(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Account not found"));
    Customer customer = customerRepository.findByAccountId(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Customer not found"));
    
    // Validar
    List<String> errors = validationService.validateAccountUpdate(request);
    if (!errors.isEmpty()) {
      throw new ValidationException(String.join(", ", errors));
    }
    
    // Actualizar
    updateAccountFromRequest(account, request);
    updateCustomerFromRequest(customer, request);
    
    // Guardar (si falla uno, rollback automático)
    accountRepository.save(account);
    customerRepository.save(customer);
    
    return Map.of("message", "Account updated successfully");
  }
}
