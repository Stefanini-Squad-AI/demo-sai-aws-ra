🔐 AUTH - Autenticação e Autorização

Guia de Desenvolvimento de User Stories

📋 Visão Geral

Propósito: O módulo AUTH é o núcleo de segurança do CardDemo. Gerencia autenticação, autorização por papéis e segurança de sessões.

Contexto de Negócio: Garante controle de acesso seguro para usuários admins e de back-office, certificando que cada papel acesse apenas as funcionalidades permitidas.

Localização do código:

  • /app/features/auth/authSlice.ts - Gerenciamento de estado (Redux)
  • /app/services/authApi.ts - Serviços de API
  • /app/components/auth/ProtectedRoute.tsx - Proteção de rotas
  • /app/hooks/useSecureSession.ts - Gestão de sessão
  • /app/pages/LoginPage.tsx - Página de login

🎯 Guía de Desarrollo de User Stories

📝 Plantillas Específicas de US

Login de Usuário

Como usuário do sistema, quero fazer login com minhas credenciais para acessar as funcionalidades conforme meu papel

Criterios de Aceitação:
  • O formulário mostra campos para User ID e Password
  • Os campos aceitam no máximo 8 caracteres
  • O texto é convertido automaticamente para maiúsculas
  • Valida-se que os campos estejam preenchidos antes do envio
  • Exibe mensagem de erro se as credenciais forem inválidas
  • Após login bem-sucedido, redireciona conforme o papel
  • Tokens são armazenados com segurança no localStorage
Simples (1-2 pts)
Proteção de Rotas por Papel

Como administrador do sistema, quero que as rotas sejam protegidas conforme papéis para garantir que os usuários acessem apenas funções autorizadas

Criterios de Aceitação:
  • As rotas exigem autenticação para acesso
  • Verifica-se o papel do usuário nas rotas restritas
  • Usuários não autenticados redirecionados ao login
  • Usuários com papel incorreto enviados ao menu certo
  • Pergunta-se a URL destino para redirecionar após o login
Médio (3 pts)
Gestão Segura de Sessão

Como usuário autenticado, quero que minha sessão expire automaticamente após inatividade para proteger minha conta contra acessos não autorizados

Criterios de Aceitação:
  • A sessão dura no máximo 8 horas
  • Verifica-se a expiração a cada 5 minutos
  • O token renova automaticamente se estiver próximo de expirar
  • Ao expirar, ocorre logout automático
  • Informações são limpas ao fechar janela/aba
  • Verifica-se a sessão ao alternar a visibilidade da aba
Médio (3-5 pts)
Renovação Automática de Token

Como sistema, quero renovar automaticamente os tokens de acesso para manter a sessão ativa sem interrupções

Criterios de Aceitação:
  • O token renova a cada 5 minutos enquanto a sessão estiver ativa
  • Utiliza o refresh token para obter novo access token
  • Se a renovação falhar, ocorre logout automático
  • A renovação não interrompe a experiência do usuário
  • O novo token substitui o anterior no localStorage
Médio (3-5 pts)

⚡ Fatores de Aceleração do Desenvolvimento

🔧 ProtectedRoute Component

Reutilização: Componente pronto para proteger qualquer rota

Uso típico:

<ProtectedRoute requiredRole="admin"> <AdminPage /> </ProtectedRoute>

Aceleração: Proteção de rotas sem escrever lógica adicional

🎣 useSecureSession Hook

Reutilização: Hook com toda a lógica de gestão de sessão

Métodos disponibles:

  • checkSessionExpiry() - Verificar se a sessão está ativa
  • attemptTokenRefresh() - Renovar token manualmente
  • clearSession() - Limpar sessão e redirecionar
  • performImmediateLogout() - Logout imediato

Aceleración: Gestión completa de sesión con una línea

📦 authSlice (Redux)

Reutilização: Redux slice completo com todas as ações

Ações disponíveis:

  • loginUser - Autenticar usuário
  • logoutUser - Encerrar sessão
  • refreshToken - Renovar access token
  • validateToken - Validar token JWT
  • immediateLogout - Logout sem confirmação

Aceleração: Estado global de auth totalmente gerenciado

🌐 authApi Service

Reutilização: Serviço com todos os endpoints de autenticação

Endpoints:

  • POST /auth/login - Login
  • POST /auth/logout - Logout
  • POST /auth/refresh - Renovar token
  • POST /auth/validate - Validar token
  • GET /auth/health - Health check

Aceleração: API client configurado e tipado

🔧 Fundamentos Técnicos

🏗️ Componentes-Chave

authSlice.ts

Localização: /app/features/auth/authSlice.ts

Responsabilidade: Gerenciar o estado de autenticação no Redux

Estado gerenciado:

  • user - Usuário autenticado atual
  • token - Access token JWT
  • isAuthenticated - Estado de autenticado
  • isLoading - Indicador de carregamento
  • error - Erros de autenticação

authApi.ts

Localização: /app/services/authApi.ts

Responsabilidade: Comunicação com endpoints de autenticação

Tipos definidos:

  • AuthResponse - Resposta do login
  • TokenValidationResponse - Resposta de validação

ProtectedRoute.tsx

Localização: /app/components/auth/ProtectedRoute.tsx

Responsabilidade: Proteger rotas segundo autenticação e papéis

Props:

  • children - Componente a proteger
  • requiredRole (opcional) - Papel requerido: 'admin' | 'back-office'

useSecureSession.ts

Localização: /app/hooks/useSecureSession.ts

Responsabilidade: Gestão segura de sessão com validações automáticas

Características:

  • Verificação de expiração (máx. 8 horas)
  • Renovação automática a cada 5 minutos
  • Limpeza ao fechar janela
  • Validação ao alternar visibilidade da aba

LoginPage.tsx

Localização: /app/pages/LoginPage.tsx

Responsabilidade: Interface do usuário para autenticação

Características:

  • Formulário com validação de campos
  • Conversão automática para maiúsculas
  • Limiar de 8 caracteres por campo
  • Toggle de visualização da senha
  • Tratamento de erros amigável
  • Redirecionamento conforme o papel

🔗 Dependências

Dependências Internas:
  • Store (Redux): Para gestão de estado global
  • Router: Para navegação e proteção de rotas
  • API Client: Para comunicação com o backend
  • Types: Interfaces compartilhadas (User, LoginCredentials)
Módulos que dependen de AUTH:
  • Account: Requer autenticação para operações
  • Credit Card: Requer autenticação e papel
  • Transaction: Requer autenticação
  • User: Requer papel admin
  • Menu: Mostra opções conforme o papel
  • Bill Payment: Requer autenticação

📋 Regras de Negócio

Autenticação

  • User ID: Máximo de 8 caracteres, obrigatório, convertido para maiúsculas
  • Password: Máximo de 8 caracteres, obrigatório, convertido para maiúsculas
  • Papéis: Admin (userType: 'A') ou Back-office (userType: 'U')
  • Tokens: Access token válido por 1 hora, refresh token válido durante a sessão

Sessão

  • Duração máxima: 8 horas contínuas
  • Verificação: A cada 5 minutos
  • Renovação: Automática se o token estiver próximo de expirar
  • Expiração: Logout automático após 8 horas ou ao fechar a janela

Autorização

  • Rotas protegidas: Requerem autenticação
  • Rotas por papel: Verificam o papel específico
  • Redirecionamento: Admin → /menu/admin, Back-office → /menu/main
  • Sem acesso: Redireciona ao login ou ao menu correspondente

🔒 Considerações de Segurança

✅ Boas Práticas Implementadas

  • Armazenamento seguro: Tokens no localStorage com limpeza total ao logout
  • Validação contínua: Verificação de token em cada rota protegida
  • Expiração automática: Sessões expiram depois de 8 horas
  • Renovação automática: Tokens renovam a cada 5 minutos
  • Logout imediato: Se o token for inválido ou expirar
  • Limpeza ao fechar: SessionStorage limpo ao fechar a janela

⚠️ Melhorias Futuras (Tech Debt)

  • Cookies HttpOnly: Migrar tokens para cookies HttpOnly para maior segurança
  • Proteção CSRF: Implementar tokens CSRF
  • Rate Limiting: Limitar tentativas de login (prevenir brute force)
  • 2FA: Autenticação de dois fatores
  • Biometric Auth: Autenticação biométrica para mobile
  • Políticas de senha: Regras mais rigorosas para senhas
